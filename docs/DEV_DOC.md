# 慢病用药小管家｜开发文档（MVP骨架）

## 1. 产品定义

### 1.1 背景问题

慢病患者线上复诊开药后，常出现：

- 忘记按时服药、服药不规律（依从性差）
- 缺少用药记录，复诊时难以回溯
- 重复购买/忘记续方导致断药
- “问诊－购药－用药－续方”服务不连贯
- 药太多/太复杂：品种多、用量不同、吃药时间点不同，容易混淆或漏服
- 药品名称与外观对不上：家属协助时更容易认错药

### 1.2 核心目标（面试展示口径）

- **不漏服/不错服**：可执行的用药计划 + 到点提醒 + 打卡记录
- **不重复买/不断药**：库存/剩余天数预测 + 续方提醒
- **可识别患者**：登录鉴权，确保“是哪位患者”的用药计划/记录可追踪并可跨设备同步（后端能力允许时）
- **低门槛录入**：药品名称/用量/库存可快速录入，支持拍照留存以减少认错药风险

### 1.3 MVP范围（按你设想的 solution 落地）

必做：

- 登录鉴权：明确是哪个患者（最小可用：邮箱/手机号+验证码，或账号密码）
- 建立用药计划（药品名称、用量、每次用量、吃药时间点、起止日期、注意事项）
- 药品库存管理（当前库存/单位/每次消耗）与“预计用尽日期”预测
- 药品图片：支持上传/拍摄药品外观图片，用于人工核对
- AI 辅助识图（MVP可先做“可插拔”能力与交互占位，先不强依赖识别准确率）
- 今日用药列表 + 快速打卡（已服/跳过/延后）
- 到点提醒（优先采用“系统日历提醒”闭环：导出/下载 `.ics`）
- 买药/续方提醒：基于库存预测的提前提醒策略（同样落在日历事件中）
- 复诊提醒：支持手动录入“医生建议复诊日期/时间”并提醒（落在日历事件中）

暂不做（后续迭代）：

- AI 解析处方/医嘱文本自动生成计划（后置）
- Web Push / 后台常驻提醒（iOS/权限/后台限制多，后置）
- 与电商/药店/医院系统的真实对接（用“跳转/提醒”模拟）

## 2. 用户体验与信息架构

### 2.1 页面

- `/`：产品介绍 + CTA
- `/plan`：用药计划列表（添加/编辑药品、时间点、注意事项）
- `/medications`：药品库（名称/用量/库存/图片），与计划关联
- `/today`：今日用药时间表 + 打卡
- `/calendar`：导出 `.ics`（用药提醒/续方提醒/复诊提醒）说明与下载
- `/followups`：复诊/随访提醒（手动录入医生建议的复诊时间）
- `/login`、`/register`：登录/注册
- `/settings`：时区、提醒提前量、数据导出/清空等

### 2.2 核心用户路径（闭环）

登录 → 录入药品（含库存/图片）→ 配置各药吃药时间点 → 生成今日时间表 → 到点提醒 → 打卡记录 → 库存/续方预测 → 买药/续方提醒 → 录入复诊时间 → 复诊提醒

## 3. 数据模型（MVP）

> MVP建议：优先实现“登录后仍可本地先跑通”，并为后续云端同步预留接口。落地路径：
>
> - 第一步（最快闭环）：登录/鉴权完成后，主要数据仍存本地（IndexedDB / localStorage），`.ics` 从本地数据生成并下载。
> - 第二步（扩展）：接入数据库后，把 `User/Medication/Plan/DoseLog` 同步到云端，实现跨设备一致。

### 3.1 结构化实体

- **User（患者/账号）**
  - `id`
  - `displayName`
  - `phone?` / `email?`
  - `timezone`

- **Medication（药品）**
  - `id`
  - `userId`
  - `name`（如“阿托伐他汀”）
  - `dosageText`（如“20mg/片”“500mg/粒”）
  - `form?`（片/粒/支/袋等，用于展示与库存单位默认值）
  - `instructions`（注意事项）
  - `images[]?`（药品外观图：URL/本地引用）
  - `aiHints?`（识别结果草稿：候选名称/置信度/备注，MVP可选）

- **Plan（用药计划/医嘱映射）**
  - `id`
  - `userId`
  - `medicationId`
  - `startDate`
  - `endDate?`
  - `times[]`（如 `["08:00","20:00"]`，每种药可不同）
  - `takenPerDose`（每次几片/几粒/几支）
  - `notes?`（医嘱/备注）
  - `timezone`

- **DoseLog（打卡记录）**
  - `id`
  - `planId`
  - `plannedAt`
  - `status`：`taken | skipped | delayed`
  - `note?`

- **Supply（库存/配药）**
  - `id`
  - `userId`
  - `medicationId`
  - `onHand`（当前库存数量）
  - `unit`（片/粒/支）
  - `refillLeadDays`（提前几天提醒，如 3/5/7）
  - `updatedAt`

- **FollowUp（复诊/随访）**
  - `id`
  - `userId`
  - `title`（如“心内科复诊”）
  - `scheduledAt`（医生建议复诊时间）
  - `remindAt?`（提前提醒时间点，可选；未填则使用全局默认）
  - `note?`（医生建议/检查项目）

### 3.2 日历提醒策略（本阶段：下载 .ics）

本阶段不做“订阅链接自动更新”。我们采用：

- 用户在 `/calendar` 点击下载 `.ics` 文件
- 导入系统日历后，由系统日历在用药时间点提醒

`.ics` 内建议包含三类事件：

- **用药事件**：按 `Plan.times[]` 生成（默认生成未来 30 天）
- **续方/买药提醒事件**：按“预计用尽日期 - refillLeadDays”生成
- **复诊提醒事件**：按 `FollowUp.scheduledAt`（或 `remindAt`）生成

后续若需要“跨设备自动更新的订阅”，再引入最小云端存储，把 `.ics` 从“导出”升级为“订阅 URL”。

## 4. API 设计（MVP草案）

> 如果 MVP 主要依赖本地数据，API 可以极简；如果要做登录鉴权与云端同步，API 需要扩展。

最小可用（支持下载导出）：

- `GET /api/ical`
  - 返回 `text/calendar`（`.ics`）
  - 生成逻辑：
    - MVP1：返回示例 `.ics`（便于先验证“提醒落地可用”）
    - MVP2：前端把 `Plan/Supply/FollowUp` 作为参数提交（或服务端从云端读取），生成未来 30 天事件并导出

鉴权（选择其一实现即可）：

- `POST /api/auth/login`（短信/邮箱验证码，或密码登录）
- `POST /api/auth/logout`
- `GET /api/auth/me`

## 5. 技术方案

### 5.1 框架

- Next.js（App Router）+ TypeScript + Tailwind
- 鉴权：NextAuth（或自建轻量 JWT/session）实现登录鉴权与用户隔离
- 数据存储：
  - MVP1：本地存储（IndexedDB 优先，localStorage 兜底）
  - MVP2：云端数据库（如 Postgres）+ Prisma，实现跨设备同步
- 日历导出：Route Handler 输出 `.ics`（下载）
- 图片：对象存储（MVP1 可先用本地/内存 URL；MVP2 使用云存储）

### 5.2 部署（面试官可直接打开）

推荐：

- Vercel 部署 Web + API
- （可选）需要跨设备同步/订阅自动更新时，引入数据库与对象存储

## 6. 里程碑与验收

### 里程碑A：骨架（当前）

- 页面结构 + 导航
- iCal 示例接口可访问
- 文档齐全

验收：

- 打开首页可看到“药品库 / 用药计划 / 今日 / 日历导出 / 登录注册入口”
- 访问 `/api/ical` 能下载一个 `.ics` 文件并导入日历看到示例事件

### 里程碑B：登录闭环（无AI，先把提醒/库存/复诊做稳）

- 登录/鉴权：能区分不同患者账号
- 药品库：名称/用量/库存/图片 本地增删改查
- 用药计划：每种药不同时间点/每次用量/起止时间，本地增删改查
- 今日时间表：根据计划生成今日剂量项
- 打卡：写入本地日志
- 库存预测：剩余天数与风险提示（预计用尽日期）
- 复诊提醒：可手动录入复诊时间并展示
- 日历导出：用药 + 续方/买药提醒 + 复诊提醒，生成 `.ics` 下载（未来 30 天）
